
1. 什么是前后端分离？

- 概念 ：前后端分离是指将用户界面（前端）和业务逻辑与数据处理（后端）分成两个独立的项目进行开发和运行。它们之间通过网络接口（如 HTTP API）进行数据交互，互不干扰。
- 在本项目中的体现 ：
  - 后端 ：使用 FastAPI 框架编写（入口为 main_server.py ），负责与数据库交互、调用大模型 API（如 Qwen）以及执行各种工具逻辑。它向外提供如 /v1/chat/ 这样的 HTTP 接口。
  - 前端 ：使用 Streamlit 框架编写（在 demo/ 目录下），它只负责页面的展示（如聊天框、侧边栏）。当用户发送消息时，前端会通过 requests.post 请求后端的接口，并将后端返回的数据渲染在页面上。前端本身不直接连接数据库或调用大模型。
2. 历史对话如何存储，以及如何将历史对话作为大模型的下一次输入？

- 历史对话的存储方式 ：
  本项目将对话数据存在了两个地方，做了职责分离：
  
  1. 用于页面展示的存储 ：在 services/chat.py 中，通过 SQLAlchemy 将用户的提问和 Agent 的回答保存到关系型数据库中（对应 ChatSessionTable 和 ChatMessageTable ）。前端查询历史记录列表时，读取的是这个数据库。
  2. 用于大模型上下文的存储 ：使用了 openai-agent 框架自带的 AdvancedSQLiteSession 模块，将会话状态单独存在了 SQLite 文件（ ./assert/conversations.db ）中。
- 如何将历史对话作为下一次输入 ：
  在代码的 chat 函数中，每次处理用户提问时，会根据当前对话的 session_id 实例化一个 session 对象（ session = AdvancedSQLiteSession(session_id=session_id, ...) ）。
  随后，在调用大模型运行器时，直接将该 session 传入： Runner.run_streamed(agent, input=content, session=session) 。
  底层框架会自动根据这个 session_id 去数据库中读取之前的历史对话记录，并和本次的新问题拼接在一起，作为完整的上下文输入给大模型，从而实现多轮对话的连续对话功能。
