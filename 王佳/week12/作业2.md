# 前后端分离与历史对话存储分析

## 什么是前后端分离？

在 06-stock-bi-agent 项目中，前后端分离的实现方式如下：

1. **后端实现**：
   - 使用 FastAPI 框架构建 RESTful API 服务
   - 提供了 `/v1/chat`、`/v1/chat/init`、`/v1/chat/get` 等接口
   - 负责处理业务逻辑、数据存储、大模型交互等核心功能
   - 运行在 `http://0.0.0.0:8000` 端口

2. **前端实现**：
   - 使用 Streamlit 框架构建交互式 Web 界面
   - 包含用户管理、股票分析、数据管理、智能问答等多个功能模块
   - 通过 HTTP 请求与后端 API 进行通信
   - 负责用户界面展示和用户交互体验

3. **分离特点**：
   - 前后端代码完全分离，独立部署
   - 前端通过 API 调用后端服务，不直接操作数据库
   - 后端提供标准化的接口，前端可以灵活调用
   - 数据流：用户操作 → 前端 → API 请求 → 后端处理 → API 响应 → 前端展示

## 历史对话如何存储，以及如何将历史对话作为大模型的下一次输入

### 历史对话存储

1. **存储方式**：
   - 使用 SQLite 数据库存储对话历史
   - 主要涉及两个数据表：
     - `ChatSessionTable`：存储对话会话信息（会话ID、用户ID、标题等）
     - `ChatMessageTable`：存储具体的消息内容（消息ID、对话ID、角色、内容等）
   - 数据库文件路径：`./assert/conversations.db`

2. **存储流程**：
   - 当用户开始新对话时，调用 `init_chat_session` 函数创建对话会话
   - 每次用户发送消息或接收回复时，调用 `append_message2db` 函数将消息存储到数据库
   - 消息包含角色（user/assistant/system）和内容

### 历史对话作为大模型的下一次输入

1. **会话管理**：
   - 使用 `AdvancedSQLiteSession` 管理对话历史
   - 每个对话都有唯一的 `session_id`，用于关联历史消息

2. **大模型输入处理**：
   - 在 `chat` 函数中，通过 `session` 参数将历史对话传递给大模型
   - 当用户发送新消息时，系统会：
     1. 首先将用户消息存储到数据库
     2. 获取系统提示（`get_init_message`）
     3. 初始化 Agent 时，将 `session` 传递给 `Runner.run_streamed`
     4. 大模型会根据历史对话上下文生成回复
     5. 将大模型的回复存储到数据库

3. **流式输出**：
   - 使用 Server-Sent Events (SSE) 技术实现实时流式输出
   - 大模型生成回复时，会实时将内容传递给前端，提升用户体验

### 核心代码实现

1. **历史对话存储**：
   ```python
   def append_message2db(session_id: str, role: str, content: str) -> bool:
       with SessionLocal() as session:
           message_recod = session.query(ChatSessionTable.id).filter(ChatSessionTable.session_id == session_id).first()
           if message_recod:
               message_recod = ChatMessageTable(
                   chat_id=message_recod[0],
                   role=role,
                   content=content
               )
               session.add(message_recod)
               session.commit()
   ```

2. **历史对话作为大模型输入**：
   ```python
   # openai-agent支持的session存储，存储对话的历史状态
   session = AdvancedSQLiteSession(
       session_id=session_id, # 与 系统中的对话id 关联，存储在关系型数据库中
       db_path="./assert/conversations.db",
       create_tables=True
   )
   
   # 流式调用大模型，传入session参数
   result = Runner.run_streamed(agent, input=content, session=session)
   ```

通过这种方式，系统能够保持对话的连贯性，让大模型能够基于历史对话内容生成更加准确和相关的回复。
